# 軟體工程師能力評估維度說明

## 📊 評估維度架構

本評估系統採用 **6 個維度** 全面評估軟體工程師的能力，涵蓋從基礎到進階、從技術到協作的各個面向。

### 1. 演算法與資料結構 (Algorithm & Data Structure)
**目標**：評估候選人的基礎計算機科學知識與問題解決能力

**評估重點**：
- 常見資料結構的選擇與應用（陣列、堆疊、佇列、雜湊表、樹、圖）
- 演算法時間與空間複雜度分析（Big O notation）
- 經典演算法問題（排序、搜尋、遞迴、動態規劃）
- 邊界條件處理與優化思維

**職級區分**：
- Entry: 認識基本資料結構，理解 O(1) vs O(n)
- Junior: 能選擇合適的資料結構解決問題
- Mid: 能分析複雜度並進行演算法優化
- Senior: 能設計高效演算法解決複雜問題
- Staff: 具備演算法創新能力，能指導團隊

---

### 2. 前後端開發 (Full-Stack Development)
**目標**：評估候選人的全端開發能力與對 Web 技術的掌握

**評估重點**：
- HTTP 協定與 RESTful API 設計
- 前端技術（HTML/CSS/JavaScript、框架使用）
- 後端開發（伺服器架構、API 開發、中介軟體）
- 即時通訊技術（WebSocket、SSE、Long Polling）
- 跨域請求處理（CORS）與瀏覽器機制

**職級區分**：
- Entry: 理解 HTTP 基礎，能使用框架開發簡單功能
- Junior: 能獨立開發前後端功能，理解 API 設計
- Mid: 能設計良好的 API 架構，處理複雜的前後端互動
- Senior: 能設計可擴展的全端架構，優化使用者體驗
- Staff: 具備技術選型能力，能引領團隊技術方向

---

### 3. 系統設計與架構 (System Design & Architecture)
**目標**：評估候選人設計可擴展、可維護系統的能力

**評估重點**：
- 系統擴展性設計（水平擴展 vs 垂直擴展）
- 分散式系統架構（負載均衡、訊息佇列、快取層）
- 微服務架構與服務拆分
- 資料一致性權衡（強一致性 vs 最終一致性、CAP 定理）
- 高可用性與容錯設計

**職級區分**：
- Entry: 理解基本的系統架構概念
- Junior: 能設計簡單的系統架構
- Mid: 能處理中等規模系統的擴展性問題
- Senior: 能設計大規模分散式系統，權衡各種架構取捨
- Staff: 能主導公司級架構演進，建立技術標準

---

### 4. 效能與監控 (Performance & Monitoring)
**目標**：評估候選人優化系統效能與建立可觀測性的能力

**評估重點**：
- 前端效能優化（圖片懶載入、程式碼分割、CDN）
- 後端效能優化（快取策略、資料庫查詢優化、N+1 問題）
- 資料庫索引設計與查詢優化
- 監控與追蹤（分散式追蹤、監控指標、日誌分析）
- 效能瓶頸排查與分析

**職級區分**：
- Entry: 認識基本的效能概念
- Junior: 能使用工具找出效能問題
- Mid: 能獨立優化常見的效能瓶頸
- Senior: 能系統性地建立效能監控與優化策略
- Staff: 能建立組織級的效能標準與最佳實踐

---

### 5. 資安與合規 (Security & Compliance)
**目標**：評估候選人的資安意識與實作安全系統的能力

**評估重點**：
- 身份驗證與授權（JWT、OAuth、Session）
- 常見安全漏洞防護（SQL Injection、XSS、CSRF）
- 密碼與敏感資料處理（加密、雜湊、加鹽）
- HTTPS 與傳輸安全
- OWASP Top 10 安全風險
- 資料保護與合規（GDPR、個資法）

**職級區分**：
- Entry: 認識基本的資安概念（如不要明文存密碼）
- Junior: 能實作基本的安全防護措施
- Mid: 能獨立設計安全的認證授權系統
- Senior: 能進行安全審計與滲透測試，建立安全機制
- Staff: 能建立組織級的資安文化與應變流程

---

### 6. 維運與協作 (DevOps & Collaboration)
**目標**：評估候選人的維運能力、團隊協作與技術決策能力

**評估重點**：
- CI/CD 流程與自動化部署
- 部署策略（藍綠部署、金絲雀部署、滾動更新）
- 生產環境事故處理與根因分析（RCA）
- 技術選型與決策
- Code Review 與團隊協作
- 技術領導與知識分享

**職級區分**：
- Entry: 理解基本的部署流程
- Junior: 能參與 CI/CD 流程，進行 Code Review
- Mid: 能獨立處理生產環境問題，參與技術決策
- Senior: 能主導事故處理，建立 SRE 實踐，影響團隊技術決策
- Staff: 能建立組織級的 DevOps 文化，跨團隊技術領導

## 📈 職級評估標準

### 整體職級定義

| 職級 | 得分率 | 特徵描述 |
|------|--------|---------|
| **Entry** | 0-40% | 具備基礎概念理解，需要指導完成任務 |
| **Junior** | 40-55% | 能在指導下獨立完成開發任務，理解實作細節 |
| **Mid** | 55-70% | 能獨立設計與實作功能，理解權衡取捨 |
| **Senior** | 70-85% | 能設計複雜系統，處理生產環境問題，影響團隊決策 |
| **Staff** | 85%+ | 具備技術領導力，能主導架構演進，建立技術標準 |

### 職級計算方式

本系統採用 **中位數法** 計算整體職級，原因如下：

1. **避免單一強項掩蓋弱項**：如果某人在「演算法」拿 95%（Staff）但其他維度都是 40%（Junior），整體職級應該是 Junior 而非 Senior
2. **平衡極端值**：中位數對極端值不敏感，能更準確反映候選人的整體水準
3. **全面能力評估**：確保候選人在多個維度都達到該職級要求

## 🎯 情境題評分指引

情境題採用分層評分，清楚標註各層級的預期答案水準：

### 評分格式範例

```
【基礎 1 分 - Entry/Junior】理解問題需求，提出基本解法
【進階 2 分 - Mid/Senior】提出系統性解決方案，考慮權衡
【專家 1 分 - Senior/Staff】考慮長期影響與組織層面改善
```

### 評分原則

1. **累進式評分**：答出基礎項才能獲得基礎分，進階項需建立在基礎之上
2. **職級對應**：
   - Entry/Junior：答出 1-2 項基礎內容（20-40%）
   - Mid：答出基礎 + 部分進階（50-70%）
   - Senior：答出基礎 + 大部分進階（70-85%）
   - Staff：答出基礎 + 進階 + 專家項（85%+）
3. **質量評估**：不僅看是否提到關鍵字，還要看解釋的深度與完整性

## 🔄 其他職級徽章互動方案（備選）

### 方案 A：Hover 顯示詳細分數
- 滑鼠移到職級徽章上顯示「🚀 Mid - 68.5%」
- 簡單直觀，不佔空間
- **推薦指數：⭐⭐⭐⭐**

### 方案 B：點擊展開各維度
- 點擊整體職級徽章，下方展開顯示各維度職級
- 「演算法: Senior ⭐, 程式設計: Mid 🚀, ...」
- 互動性強，資訊完整
- **推薦指數：⭐⭐⭐⭐⭐**

### 方案 C：表格行下方顯示小型雷達圖
- 每筆記錄可展開，顯示該候選人的能力雷達圖
- 視覺化程度最高
- 需要更多開發工作
- **推薦指數：⭐⭐⭐**

### 🎖️ 建議實作方案

**採用 方案 B（點擊展開）+ 方案 A（Hover 提示）的組合**：
- 預設顯示整體職級徽章（如「🚀 Mid」）
- Hover 顯示分數（「Mid - 68.5%」）
- 點擊徽章展開顯示各維度職級與分數
- 兼具簡潔性與完整性

## 📊 獨立職級分析頁面規劃（備選）

### 頁面內容建議

1. **整體職級分布圓餅圖**
   - 顯示 Entry/Junior/Mid/Senior/Staff 的人數分布
   - 幫助了解整體水準

2. **各維度平均職級長條圖**
   - 橫向長條圖顯示每個維度的平均職級
   - 找出團隊的強項與弱項

3. **個別候選人職級雷達圖**
   - 可多選候選人進行對比
   - 6 個維度的職級分布（Entry=1, Junior=2, ..., Staff=5）

4. **職級成長建議**
   - 根據候選人的弱項維度自動生成學習建議
   - 例如：「資安維度為 Entry，建議學習 OWASP Top 10」

5. **職級趨勢分析**
   - 如果有歷史資料，顯示職級隨時間的變化
   - 追蹤學習成長軌跡

### 實作優先級

- **P0（必須）**：整體職級分布圓餅圖
- **P1（重要）**：各維度平均職級長條圖
- **P2（建議）**：個別候選人職級雷達圖對比
- **P3（未來）**：職級成長建議與趨勢分析

## 🔧 題庫維護指南

### 新增題目原則

1. **維度平衡**：確保每個維度都有 Entry 到 Senior 的題目
2. **難度梯度**：避免某個職級的題目過於集中
3. **語言風格**：中文為主，專有名詞用括號標註英文
4. **實務導向**：題目應反映真實工作場景

### 題目審核清單

- [ ] 題目描述清晰，無歧義
- [ ] 選項設計合理，有一定迷惑性
- [ ] 正確答案明確
- [ ] Explanation 詳細完整，說明為何其他選項錯誤
- [ ] 專有名詞正確標註英文
- [ ] maxScore 與難度相符
- [ ] 情境題的 rubric 清楚標註職級與分數

### 定期維護

- **每季審查**：根據候選人反饋調整題目難度
- **半年更新**：新增 5-10 題，淘汰過時或過簡單的題目
- **年度大改版**：根據技術趨勢調整評估重點
